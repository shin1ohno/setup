# コンテンツ鮮度管理

このリポジトリはドキュメント・ハブであり、保持する情報の大半は外部（Notion、Slack、GitHub、Jira、外部 Web）への参照リンクです。これらは時間とともに必ず劣化します。劣化に備えた書き方を全員（人間 + AI）に適用します。

## 日付注記の義務

外部リンクを文書に書く際は、以下のいずれかの方法で「いつ時点の内容か」を残します:

- インライン: `[ページタイトル](URL)（2026-05 時点）`
- 表の隣接カラム: `参照日` 等の列を設ける
- セクション末尾: `※ このセクションは 2026-05-18 に整理`

日付の粒度は文書の更新頻度に合わせる:

- 日次〜週次で動く情報（プロジェクトステータス、進行中の議論）→ `YYYY-MM-DD`
- 月次〜四半期で動く情報（パートナー一覧、スケジュール）→ `YYYY-MM`
- 年単位で動く情報（背景、目的）→ `YYYY`

## 鮮度マーカーが必要な対象

- Notion ページのリンク（ページ内容は逐次更新される）
- Slack のメッセージ・スレッドリンク（コンテキストが流れる）
- GitHub Issue / PR の状態言及（クローズ・マージで意味が変わる）
- 外部 SaaS のステータス記述（料率、人員、規模感）
- 数値・規模感（「約 21,000 店舗」「8 月にジャッジ」等）

## 鮮度マーカーが不要な対象

- 不変リンク: 公開仕様、リリース済バージョンのドキュメント、permalink
- 一般原則・規律（このファイル自身など）

## 更新フロー

README / CLAUDE.md を更新するとき:

1. 既存セクションの日付注記を確認
2. 該当情報が現在も真か簡易確認（Notion/Slack/`gh` で 1〜2 件 probe）
3. 真なら日付を更新、誤なら情報自体を修正
4. 確認できないなら情報を残しつつ「未確認 = `YYYY-MM-DD` 時点では真、以降未確認」と明記

## なぜ必要か

ドキュメント・ハブの存在価値は「ここを見れば最新が分かる」ことです。古い情報が「いつ古いか」分からないまま残ると、ハブ全体の信頼性が崩れ、メンバーは結局 Notion / Slack を直接探すようになり、ハブの存在意義が消えます。

日付注記は、古さを可視化する最小コストの仕組みです。
