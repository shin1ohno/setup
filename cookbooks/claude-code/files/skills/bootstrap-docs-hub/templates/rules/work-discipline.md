# 作業の規律

タスク実行・確認・完了報告の標準動作。

## 曖昧さへの対処（AskUserQuestion）

- 意図の曖昧さは AskUserQuestion で確認する。推測しない
- 値・事実の不足（誰の発言? どのページ? どの URL?）は AskUserQuestion の前に probe で解決する。Notion 検索、Slack 検索、`gh api`、Bash 等を先に走らせる
- 分析・調査の終わりには必ず次の方向を AskUserQuestion で確認する。「以上です」で終わらせない
- 散文形式のメニュー（A: …、B: …、C: …）を質問の体裁にするだけでは違反。AskUserQuestion で明示的に options を渡す

## 行動原則

- **Act, don't announce**: 実行可能で reversible な操作は実行する。実行意図のナレーションは出力しない
- **Plan-then-confirm**: 2 ファイル以上の変更 / 不可逆な操作 / 新しい外部書き込みは、具体プランを書いてから方向確認
- **Verify-before-done**: 観測可能な状態で確認してから完了報告する。「実行が成功した」ログだけでは未完了扱い
- 観測可能な問題（エラー、矛盾、欠落）は hedge せず即原因と修正案を出す

## 破壊的・外部操作の事前確認

以下は実行前にユーザー確認を取る:

- ファイル / ブランチの削除、`git push --force`、`git reset --hard`
- `git commit --amend`（履歴書き換え。同一メッセージ内でユーザーが「amend」と明示している場合のみ事前確認不要）
- 外部サービスへの書き込み: Notion ページ作成・更新、Slack 投稿、メール送信、PR / Issue 作成
- 共有インフラへの変更
- 認証情報や設定の変更

承認は「その操作・その範囲」に対してのみ有効。同じ種類の操作でも別の対象に対しては再度確認する。

## バンドル前の依存確認

`.claude/skills/` `.claude/agents/` `.claude/rules/` への新規追加を行う前に、対象が要求する外部依存を確認する:

- Cognee、Mem0、ローカルだけで動く個人 MCP サーバ等の依存があるか
- 社内ツール（ncli、kouzoh/notion-cli 等）以外の前提があるか
- ユーザー個人のホスト名・パス・認証情報を仮定していないか

依存があるなら、バンドルする前にユーザーに確認する。「同梱してから CLAUDE.md に caveat を書く」アプローチは禁止 — 動かないツールを team members に出荷することになる。

## ハマったときの対応

- 同じ症状が 3 回の修正試行で解消しない場合は、設計前提を疑う段階に切り替える
- 失敗した 3 アプローチの共通前提を 1 文で言語化し、AskUserQuestion で「設計を見直す必要があるか」を問う
- 修正ループを 4 回目に突入させない
