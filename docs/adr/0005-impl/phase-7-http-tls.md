# Phase 7-tls — HTTP layer TLS for Elasticsearch + Kibana

**Status**: DEFERRED — must be applied after Phase 3b (ES + Kibana cookbooks created)
and Phase 6 (Loki cutover) have merged. This branch (`feat/adr0005-phase7-http-tls`)
contains documentation + a patch file only; the cookbook files this Phase edits do
not exist yet on this branch's tree.

## Pre-conditions

1. **Phase 3b merged** (`feat/adr0005-phase3b-es-kibana-cookbooks`) — cookbooks
   `lxc-elasticsearch` + `lxc-kibana` exist with transport TLS enabled, HTTP TLS
   disabled, basic auth on HTTP. CA + node certs already fetched from SSM and
   installed under `/data/elasticsearch/certs/` (CT 112/113/114).
2. **Phase 6 merged** (`feat/adr0005-phase6-cutover-loki`) — Loki removed,
   Vector → ES sink is the sole log path. HTTPS cutover therefore affects only
   the live ES cluster, not a parallel Loki path.
3. **Phase 4 already shipped** `tls.verify_certificate = true` and
   `verify_hostname = true` in `vector.toml`. Those settings became no-ops while
   ES HTTP was plain — Phase 7 makes them effective.

## Specification

Re-use Phase 3b's transport-layer node cert (CA + per-node `<NODE_NAME>.{crt,key}`
already at `/usr/share/elasticsearch/config/certs/`). No new SSM fetches, no new
Terraform resources. Edit YAML templates only.

### Adversarial #5 — SAN audit before apply

The transport TLS cert was generated by Phase 1b with `subject_alternative_names`
covering `DNS:es-{0,1,2}.home.local`, `IP:192.168.1.<n>`, `DNS:localhost`,
`IP:127.0.0.1`. HTTP TLS reuses the same cert; if SAN is missing any hostname
clients use, Vector / Kibana / curl will fail hostname verification.

Verification before `mitamae apply`:

```bash
ssh root@es-0 'openssl x509 -in /data/elasticsearch/certs/es-0.crt -text -noout' \
  | grep -A1 'Subject Alternative Name'
# Expect: DNS:es-0.home.local, DNS:localhost, IP:192.168.1.<n>, IP:127.0.0.1
```

If any DNS name a consumer dials is absent, regenerate certs in Phase 1b first
(do NOT proceed with Phase 7-tls).

## Concrete diffs

See `phase-7-http-tls.patch` for unified diff. Summary:

1. `cookbooks/lxc-elasticsearch/files/elasticsearch.yml.tmpl`:
   - `xpack.security.http.ssl.enabled: false` → `true`
   - Add three keys: `key`, `certificate`, `certificate_authorities` — paths
     identical to the existing transport TLS keys (re-use, no new files).

2. `cookbooks/lxc-kibana/files/kibana.yml.tmpl`:
   - `elasticsearch.hosts`: ensure `https://es-{0,1,2}.home.local:9200` (Phase 3b
     plan should already have set this; the patch enforces it idempotently).
   - Add `elasticsearch.ssl.certificateAuthorities: ['/data/kibana/certs/ca.crt']`
   - Add `elasticsearch.ssl.verificationMode: full` (full hostname match — SAN
     audit above guarantees this works).

3. `cookbooks/lxc-elasticsearch/default.rb`: no change (cert install already done).
4. `cookbooks/lxc-kibana/default.rb`: no change (CA already fetched in Phase 3b).
5. `cookbooks/lxc-monitoring/files/vector.toml`: no change — Phase 4 already set
   `tls.verify_certificate = true, verify_hostname = true`. They become effective
   the moment ES HTTP TLS turns on.
6. **Out of scope**: Kibana's own browser-facing HTTPS (`server.ssl.enabled`).
   Defer to a later PR — this Phase covers ES ↔ {Vector, Kibana} only.

## Verify steps (Claude-runnable, post-apply)

```bash
# TLS handshake + cert chain verification (run from any LAN host with /tmp/ca.crt)
openssl s_client -connect es-0.home.local:9200 -CAfile /tmp/ca.crt < /dev/null 2>&1 \
  | grep -E 'Verify return code|subject='
# Expect: Verify return code: 0 (ok)

# HTTP-layer auth still works over TLS
curl --cacert /tmp/ca.crt -u "elastic:$ES_PASSWORD" -I https://es-0.home.local:9200/
# Expect: HTTP/1.1 200 OK

# ES log confirms HTTPS bind
ssh root@es-0 'docker logs elasticsearch 2>&1 | grep -i "publish_address.*9200"' | tail -3
# Expect: "publish_address {https://...:9200}"
```

## Rolling apply procedure (user-authorize)

ES cluster: never restart >1 node simultaneously; wait for cluster green
between nodes.

```bash
# 1. SAN audit (per Adversarial #5)
! ssh root@es-0 'openssl x509 -in /data/elasticsearch/certs/es-0.crt -text -noout' \
  | grep -A1 'Subject Alternative Name'

# 2. Apply per-node, waiting for cluster green between each
! sudo ./bin/mitamae local pve/lxc-es-0.rb
! curl --cacert /tmp/ca.crt -u "elastic:$ES_PASSWORD" \
    "https://es-1.home.local:9200/_cluster/health?wait_for_status=green&timeout=60s"
! sudo ./bin/mitamae local pve/lxc-es-1.rb
! curl --cacert /tmp/ca.crt -u "elastic:$ES_PASSWORD" \
    "https://es-0.home.local:9200/_cluster/health?wait_for_status=green&timeout=60s"
! sudo ./bin/mitamae local pve/lxc-es-2.rb
! curl --cacert /tmp/ca.crt -u "elastic:$ES_PASSWORD" \
    "https://es-0.home.local:9200/_cluster/health?wait_for_status=green&timeout=60s"

# 3. Kibana applies last (depends on ES HTTPS being live on all 3 nodes)
! sudo ./bin/mitamae local pve/lxc-kibana.rb
```

Yellow status between node restarts is expected (replica unassigned during the
restarting node's window). Green is the gate to proceed to the next node.

Failure mode if SAN missing a hostname: Vector / Kibana log
`tls handshake failed: x509: certificate is valid for X, not Y`. Roll back
elasticsearch.yml change on the affected node, regenerate certs in Phase 1b
with the missing DNS name, redeploy.
